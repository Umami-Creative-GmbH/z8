# CASL v7 Query Authorization Design

## Context

Z8 has upgraded `@casl/ability` to v7. The new release improves how CASL converts prioritized rules, including `cannot(...)` rules, into database query conditions. This matters for Z8 because authorization currently has two paths:

- Runtime checks through `ability.can(...)` and `ability.cannot(...)`.
- Handwritten Drizzle filters and helper functions that decide which records a user may query.

Those paths can drift. CASL v7 makes it practical to move more of Z8's authorization model into one condition-aware rule system and derive database filters from the same rules used for runtime checks.

## Goals

- Use CASL v7's priority-preserving condition behavior for database-backed authorization.
- Keep every query organization-scoped by default.
- Reduce duplicated role, self-service, manager, and team authorization logic.
- Preserve existing string-subject checks while migrating to object subjects.
- Add tests that prove runtime decisions and generated query filters enforce the same intent.

## Non-Goals

- Do not rewrite the entire authorization system in one change.
- Do not introduce Prisma or `@casl/prisma`; Z8 uses Drizzle.
- Do not remove existing route-level checks until replacement filters are covered by tests.
- Do not encode tenant-specific settings in environment variables.

## Recommended Direction

Use a phased hybrid of two approaches:

- Build a small CASL-to-Drizzle query adapter first.
- Then migrate duplicated authorization paths subject-by-subject toward conditional CASL rules.

This gives Z8 the safety of a narrow foundation while moving toward a broader authorization cleanup.

## Architecture

### Ability Model

`apps/webapp/src/lib/authorization/ability.ts` should keep supporting existing string subjects such as `ability.can("manage", "TimeEntry")`. New database-backed checks should add object subjects with explicit subject typing through CASL's `subject(...)` helper.

Rules should gradually move scoping into conditions where the scope is representable by record columns:

- `organizationId` for tenant isolation.
- `employeeId` for self-service time, absence, and report access.
- `teamId` for team-scoped permissions.
- `id` or `employeeId` sets for direct reports loaded into `PrincipalContext.managedEmployeeIds`.
- Approval fields that already exist in query models, such as requester or approver identifiers, where available.

Platform admins may keep broad `manage all` access, but org-scoped helper APIs should still require explicit organization context before returning records.

### Query Adapter

Add an internal module under `apps/webapp/src/lib/authorization/query` that converts CASL v7 condition trees into Drizzle predicates for a known subject/table mapping.

The adapter should be deliberately small and explicit:

- Accept an `AppAbility`, action, subject type, and table mapping.
- Use CASL v7 extra helpers such as `rulesToCondition` or `rulesToAST` to preserve rule priority.
- Convert supported condition operators to Drizzle predicates.
- Fail closed for unsupported operators, unknown fields, or missing table mappings.

The first supported operators should match the conditions Z8 actually needs:

- Equality.
- `$in` for managed employee IDs or team IDs.
- `$ne` only if a migration genuinely needs it.
- Boolean `and`, `or`, and `not` composition generated by CASL v7.

Avoid making this a generic ORM abstraction. It should know the fields it can translate for each subject.

### Initial Subject Rollout

Start with subjects that have high duplication and clear table fields:

- `Employee`: self, direct reports, and organization admin access.
- `TimeEntry`: own entries, direct-report entries, and workforce admin access.
- `Absence`: own requests, direct-report approvals, and admin access.
- `Approval`: approver/manager scopes and admin access.
- Reports accessible employees: replace custom accessible-employee filtering where practical.

Each subject migration should keep the previous behavior until tests prove equivalence or intentionally document a behavior correction.

## Data Flow

1. `getPrincipalContext()` loads session, active organization, membership, employee, team permissions, and managed employee IDs.
2. `defineAbilityFor(principal)` builds both existing broad string rules and new condition-aware object rules.
3. Runtime checks call `ability.can(action, subject(subjectType, record))` for record-specific decisions.
4. Query code calls the authorization query adapter to derive a Drizzle `where` predicate for the same action and subject.
5. Route and service code combines the generated predicate with feature-specific filters such as date ranges, statuses, pagination cursors, and search filters.

## Error Handling And Security

- Missing ability, missing active organization, or missing subject mapping returns no records or a 403, depending on the call site.
- Unsupported CASL conditions must fail closed rather than widening access.
- Generated predicates must always include or be combined with `organizationId` for organization-scoped records.
- Query helpers should expose narrow, typed APIs so callers cannot pass arbitrary field names from user input.
- API handlers should keep existing authentication checks before creating abilities or filters.

## Testing

Add tests at three levels:

- Ability tests for condition-aware object checks, including self, manager, admin, and cross-organization denial.
- Query adapter tests that verify translated Drizzle predicates for supported operators and fail-closed behavior.
- Feature tests for migrated routes/helpers, especially reports, absences, time entries, and approvals.

Include a CASL v7 priority regression case:

```ts
can("read", "Example")
cannot("read", "Example", { private: true })
can("read", "Example", { ownerId: principal.employee.id })
```

The test should prove that runtime checks and generated query filters both allow the user's own private record while denying another user's private record.

## Migration Plan Shape

The implementation plan should avoid a single broad rewrite. It should sequence work as:

1. Add typed subject helpers and condition-aware ability tests.
2. Add the CASL-to-Drizzle adapter and unit tests.
3. Migrate `Employee` and report-access helpers first.
4. Migrate `TimeEntry` reads.
5. Migrate absence and approval query paths.
6. Remove duplicated helper logic only after migrated callers and tests are stable.

## Open Decisions Resolved

- The project should pursue the broader option-3 direction, but through the safer option-1 adapter foundation.
- Drizzle remains the database integration target.
- Existing string-subject checks remain during migration.
- The first implementation should prioritize correctness and test coverage over a wide migration surface.
